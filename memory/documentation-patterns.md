## 模式 1：命令引用规范

### 规则
- **所有命令必须在 commands.md 定义后，才能在其他文档引用**
- 路径统一使用 `../siyuan-skill/siyuan.js`（从 llm-wiki/references/ 目录引用）
- 禁止假设命令存在——必须实际验证

### 检查清单
- [ ] 新增命令 → 先更新 commands.md → 再更新其他文档
- [ ] 删除命令 → 全局搜索替换 → 标记为 deprecated
- [ ] 修改参数 → 同步更新所有示例

## 模式 2：文档分层职责

### 职责划分

| 文档 | 职责 | 读者 | 禁止内容 |
|------|------|------|----------|
| **SKILL.md** | 技能定义、命令索引 | Claude | 详细参数、长示例 |
| **README.md** | 项目介绍、架构理念 | 新用户 | 与 SKILL.md 重复 |
| **commands.md** | 命令参数详解 | 使用者 | 未实现的命令 |
| **quick-start.md** | 快速上手示例 | 新用户 | 高级参数 |
| **ingest-guide.md** | Ingest 流程详解 | 使用者 | 其他工作流 |
| **query-patterns.md** | 查询模式 | 使用者 | 未实现的 SQL |
| **troubleshooting.md** | 故障排除 | 使用者 | 未验证的修复 |

### 重复消除原则
- README 和 SKILL.md 不应有大量重复
- 速查表 → 指向详细文档链接
- 示例代码 → 保持最小可运行

## 模式 4：验证机制

### 发布前检查（手动）

```bash
# 1. 检查是否有未定义的命令
grep -n "siyuan.js [a-z-]*" references/*.md | grep -v "$(cat commands.md | grep '^## ' | sed 's/## //')"

# 2. 检查路径一致性
grep -n "node siyuan.js" references/*.md SKILL.md README.md

# 3. 检查链接有效性
# 手动点击所有相对链接
```

## 模式 5：变更流程

### 新增命令流程
1. 在 siyuan-skill 实现并测试命令
2. 更新 commands.md（详细参数）
3. 更新 SKILL.md（命令索引）
4. 更新相关指南（示例）
5. 验证所有示例可执行

### 删除命令流程
1. 全局搜索命令引用
2. 标记为 deprecated（保留一版本）
3. 更新所有文档移除示例
4. 提供替代方案

## 经验总结

### 根因分析
文档质量问题源于：**文档编写与代码实现分离**
- 写文档时假设了某些功能存在
- 未及时同步命令变更
- 缺乏验证机制

### 系统性改进
1. **命令先行**：命令实现 → 文档更新
2. **单点真理**：commands.md 是命令定义的权威来源
3. **增量验证**：每次变更后验证相关示例
4. **模式沉淀**：将规则写入本文档，形成飞轮
