# 全自动模式详解

> **核心原则**：减少 Human-in-the-Loop，90% 场景无需确认。

## 模式对比

| 模式 | 命令 | 适用场景 | 确认次数 |
|------|------|----------|----------|
| **全自动** | `/wiki <docId>` | 日常摄入，信任 LLM 判断 | 0 次（异常除外） |
| **交互式** | `/wiki ingest <docId> -i` | 复杂文档，首次学习 | 3-5 次 |
| **手动** | 底层 siyuan-skill | 精细控制 | 每步确认 |

## 全自动执行逻辑

```
┌─────────────────────────────────────────────────────────┐
│                  全自动 Ingest 流程                       │
├─────────────────────────────────────────────────────────┤
│ Step 1: 读取文档                                         │
│   - 一次读取完整内容                                      │
│   - 失败 → 报错退出（思源未启动）                          │
├─────────────────────────────────────────────────────────┤
│ Step 2: 创建摘要页 → wiki/sources/                       │
│   - 每个来源必须有摘要页                                  │
│   - 内容：标题、来源、核心观点、关键实体链接               │
├─────────────────────────────────────────────────────────┤
│ Step 3: 提取实体                                         │
│   - 自动识别：人名、概念、工具、组织                       │
│   - 数量 ≤15 → 继续                                      │
│   - 数量 >15 → 警告但继续（建议拆分）                      │
├─────────────────────────────────────────────────────────┤
│ Step 4: 冲突检测                                         │
│   - 无冲突 → 继续                                        │
│   - 有同名实体 → 自动追加描述（不新建）                    │
│   - 有矛盾信息 → 记录到 memory/conflicts.md，继续         │
├─────────────────────────────────────────────────────────┤
│ Step 5: 批量创建/更新实体页 → wiki/entities/              │
│   - 已存在 → 追加描述块（不重写整页）                      │
│   - 不存在 → 创建新实体页                                 │
│   - 并行创建，单个失败 → 记录错误，继续其他                │
├─────────────────────────────────────────────────────────┤
│ Step 6: 更新主题页 → wiki/topics/（条件触发）             │
│   - 内容匹配已有主题 → 追加相关实体链接                   │
│   - 无匹配主题 → 跳过（不强制创建）                        │
├─────────────────────────────────────────────────────────┤
│ Step 7: 更新索引                                         │
│   ⚠️ 必须用 update 重写整页，禁止用 bi                   │
│   - 读取当前 index 内容                                   │
│   - 合并新实体到已有分类（不新建独立章节）                 │
│   - 追加源文档索引到末尾                                   │
│   - 更新统计数字                                          │
│   - 执行后验证：无裸 docId、无 prepend                    │
├─────────────────────────────────────────────────────────┤
│ Step 8: 追加日志                                         │
│   ⚠️ 必须用 update 重写整页，禁止用 bi                   │
│   - 读取当前 log 内容                                     │
│   - 在开头插入新日志条目（时间倒序）                       │
│   - 格式：## YYYY-MM-DD | ingest | 描述                  │
│   - 执行后验证：无裸 docId                               │
├─────────────────────────────────────────────────────────┤
│ 输出: "已摄入 X 个实体，更新 Y 页"                        │
└─────────────────────────────────────────────────────────┘
```

**单次摄入影响页面数**：理想情况 6-15 页（摘要 + 实体 + 主题 + index + log）

## 命令选择矩阵（重要）

> **关键规则**：不同文档类型使用不同的更新命令，选错会导致结构混乱。

| 文档类型 | 更新方式 | 命令 | 原因 |
|---------|---------|------|------|
| **index** | 重写整页 | `update` | 需要合并到已有分类，禁止 prepend |
| **log** | 重写整页 | `update` | 保持格式统一，避免裸 docId |
| **实体页** | 追加块 | `bi` | 块级追加新描述 |
| **主题页** | 追加块 | `bi` | 追加相关实体链接 |
| **摘要页** | 创建新页 | `create` | 每个来源一个独立页面 |

**禁止操作**：
- ❌ 用 `bi` 更新 index/log → 导致 prepend 和裸 docId
- ❌ 用 `update` 更新实体页 → 覆盖已有描述

**正确示例**：
```bash
# 更新 index（读取当前内容 → 合并新实体 → 重写）
node ../siyuan-skill/siyuan.js content <indexDocId>
# ... 合并内容 ...
node ../siyuan-skill/siyuan.js update <indexDocId> "完整合并后的内容"

# 追加实体描述
node ../siyuan-skill/siyuan.js bi <entityDocId> "## 来自《新来源》\n新描述..."
```

## 异常处理规则

| 异常类型 | 自动处理方式 | 是否打断用户 |
|----------|-------------|-------------|
| 文档读取失败 | 报错退出 | ❌ 只报告 |
| 实体 >15 个 | 警告继续 | ❌ 只报告 |
| 同名实体存在 | 追加描述 | ❌ 自动合并 |
| 检测到矛盾 | 记录到 conflicts.md | ❌ 只报告 |
| 单个创建失败 | 记录错误继续 | ❌ 只报告 |
| 全部创建失败 | 回滚报错 | ✅ 需人工 |
| API 连续失败 3 次 | 等待重试 | ✅ 需人工 |

## 冲突自动合并策略

```
检测到同名实体 "Neural Computer" 已存在：
  ↓
对比新内容与已有内容：
  - 完全相同 → 跳过（幂等）
  - 有新信息 → 追加到已有页面末尾
  - 有矛盾 → 记录到 conflicts.md，保留两版
  ↓
结果: 页面已更新 / 跳过 / 记录矛盾
```

## 批量摄入（多个文档）

```bash
# 批量全自动摄入
/wiki ingest doc1 doc2 doc3

# 输出
✅ doc1: 5 实体 ✓
✅ doc2: 3 实体 ✓
⚠️ doc3: 矛盾检测，已记录
━━━━━━━━━━━━━━━━━━
总计: 8 实体，1 矛盾待处理
```
