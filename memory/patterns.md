## bi 命令链接插入陷阱

**类别**: 思源操作
**日期**: 2026-04-09

**问题描述**:
使用 `siyuan.js bi` 命令插入包含链接列表的内容时，如果内容以 `((docId` 开头，思源 API 会将其解析为普通文本块而非链接语法，导致文档中出现孤立的 docId 文本（如 `<ORPHAN_DOC_ID>`）。

**根本原因**:
- `bi` 创建的是独立 block，block 级解析和文档级解析上下文不同
- 以 `((` 开头的内容在 block 边界处无法正确解析为链接

**解决方案**:
1. **优先使用 `update` 重写整个文档**，而非 `bi` 追加
2. 如必须用 `bi`，确保内容以 `- ` 或 `## ` 等 markdown 标记开头：
   ```markdown
   - ((id1 '标题1'))
   - ((id2 '标题2'))
   ```
3. 插入后**立即验证**：运行 `validate_ingest.py --doc-id <id>`

**如何应用**:
- 触发条件：需要向文档追加包含链接的内容
- 应对措施：改用 `update` 或确保内容以 markdown 列表格式开头

## Log 格式合规性

**类别**: 格式规范
**日期**: 2026-04-09

**问题描述**:
Log 文档条目格式不统一，缺少必需字段，导致难以追踪操作历史。

**标准格式**:
```markdown
## YYYY-MM-DD | 操作类型 | 描述

- 新建实体：XXX、YYY（共N个）
- 更新实体：AAA、BBB（如有）
- 触及：N 页
```

**操作类型枚举**: `ingest`, `query`, `lint`, `cleanup`, `update`

**如何应用**:
- 触发条件：每次 ingest/query/lint 后追加 log
- 应对措施：使用模板，确保包含所有必需字段

## 实体约束超限

**类别**: 认知边界
**日期**: 2026-04-09

**问题描述**:
单次处理过多实体（>4个）导致认知过载，走捷径、出错、遗漏。

**解决方案**:
1. 使用 `entity_check.py` 自动拆分
2. 每批处理后强制元反思检查点
3. 超限任务使用 TaskCreate 并行处理

**如何应用**:
- 触发条件：需要处理多个实体（文章、实体页、主题页等）
- 应对措施：运行 `entity_check.py --items "a,b,c,d,e"` 自动拆分

## 链接格式验证

**类别**: 格式规范
**日期**: 2026-04-09

**问题描述**:
链接格式错误导致无法正确写入思源 refs 图谱。

**正确 vs 错误**:
```markdown
✅ ((<ENTITY_DOC_ID> 'RAG'))
❌ ((<ENTITY_DOC_ID> RAG))      ← 标题缺少单引号
❌ ((RAG))                             ← 缺少 docId
❌ [[RAG]]                             ← Obsidian 格式
❌ <ENTITY_DOC_ID>              ← 裸 docId
```

**如何应用**:
- 触发条件：创建或修改包含链接的文档
- 应对措施：使用 `validate_ingest.py` 检查链接格式

## 后置验证缺失

**类别**: 质量保证
**日期**: 2026-04-09

**问题描述**:
写入文档后没有立即验证，导致问题发现滞后。

**解决方案**:
每次写入后执行验证：
```bash
# 验证特定文档
python scripts/validate_ingest.py --doc-id <id>

# 验证所有核心文档
python scripts/validate_ingest.py
```

**检查项**:
- 残留文本块（孤立的 docId）
- Log 格式合规性
- Index 结构完整性
- 实体页标准章节

**如何应用**:
- 触发条件：任何文档写入操作后
- 应对措施：强制执行后置验证

## Index 结构性破坏（批量摄入 prepend 问题）

**类别**: 认知边界 / 结构完整性
**日期**: 2026-04-10

**问题描述**:
批量摄入多篇文章时，LLM 将新的专题内容 prepend 到 index 页面顶部，导致：
1. 主标题 "# LLM Wiki 索引" 被挤到页面中间
2. 源文档重复列出（出现"源文档索引 - XXX 专题"、"XXX 文章集合"、"源文档索引"三处）
3. 实体索引和源文档索引的层级关系混乱

**根本原因**:
- SKILL.md 缺少 index 结构不变量约束
- LLM 面对大量新内容时选择"快速 prepend"而非"合并到已有结构"
- `validate_ingest.py` 不检查标题位置和重复条目

**解决方案**:
1. **禁止 prepend**：index 更新必须用 `update` 重写整个文档，不能用 `bi` 追加到顶部
2. **合并而非新建**：新实体归入已有分类（如"LLM/Agent 技术"），新源文档追加到"源文档索引"末尾
3. **更新后立即验证**：`python scripts/validate_ingest.py --type index`

**Index 结构不变量**:
```
# LLM Wiki 索引          ← 必须是文档第一行
## 统计
## 实体索引（按类别）    ← 新实体合并到已有分类或新建子分类
  ### [已有分类]         ← 按领域分组
  ### [新专题]           ← 大专题可作为新 ### 节
## 主题索引
## 源文档索引            ← 所有源文档合并到此处，按时间排序
```

**如何应用**:
- 触发条件：批量摄入 ≥2 篇文章后更新 index
- 应对措施：读取当前 index → 合并新内容到正确位置 → `update` 重写 → 验证

## 幂等性设计

**类别**: 完整性
**日期**: 2026-04-09

**问题描述**:
重复执行 ingest 导致重复数据，无法安全重试。

**解决方案**:
1. 使用状态文件记录已处理内容
2. 计算内容哈希，相同内容自动跳过
3. 支持 `--force` 强制重新处理

**如何应用**:
- 触发条件：设计 ingest 命令时
- 应对措施：确保实现幂等性，支持断点续传

## 知识孤岛陷阱

**类别**: 元反思

**问题**:
修复过的问题为什么又犯？规则存在于 memory/patterns.md，但执行时没有读取。

**表现**:
- 修复经验存放在 memory 中
- Skill 主文档（SKILL.md、auto-mode.md）没有同步
- 执行时只读主文档，不读 memory
- 导致"修复过但再犯"

**根因分析**:
```
知识存放位置 ≠ 知识执行位置
memory/patterns.md（存储）vs SKILL.md（执行）
```

**解决方案**:
1. **同步到主文档**：关键规则必须出现在 SKILL.md 和 auto-mode.md
2. **添加强制检查点**：在执行流程中加入 ⚠️ 警告和验证步骤
3. **执行前验证**：对照 memory/patterns.md 检查命令选择是否正确

**如何应用**:
- 触发条件：发现修复过的问题再犯
- 应对措施：
  1. 检查规则是否只在 memory 中
  2. 同步到 SKILL.md 和执行文档
  3. 添加强制检查点

**案例**:
2026-04-12：index/log 更新使用 bi 命令导致 prepend 和裸 docId
- 修复经验已在 memory/patterns.md
- 但 auto-mode.md 的 Step 7 没有明确命令选择
- 执行时选了 bi 命令
- 修复：添加"命令选择矩阵"到 auto-mode.md + SKILL.md
